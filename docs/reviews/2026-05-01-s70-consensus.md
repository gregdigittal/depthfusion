Consensus reached after Round 1+ Claude consolidation pass. No findings remained disputed at the MEDIUM+ threshold. `code-decisions-debate` was NOT invoked.

### Fixes applied (this commit)

1. **`fcntl.flock` around RMW** (X1, the keeper) — `_tool_set_memory_score` now opens a sidecar lock file (`.<filename>.scorelock`), takes `LOCK_EX` for the entire read-parse-splice-write-replace cycle, and releases on exit. This serializes concurrent calls — partial-update writers cannot race the read phase. Atomic rename retained for torn-write protection. The lock file is intentionally separate from the target file so the os.replace inode swap doesn't invalidate the lock.
2. **Type validation at boundary** (C1 / X2) — added explicit `isinstance(val, (int, float))` check before constructing `MemoryScore`. String-typed input returns `{ok: false, error: "set_memory_score: 'importance' must be a number, got str"}`.
3. **Format consistency** (C3 / C4) — both `decision_extractor.write_decisions` and `negative_extractor.write_negatives` now emit `salience: 1.0000` (was `salience: 1.0`) using `:.4f` formatting. New test `test_extractor_emits_byte_equivalent_format_to_splice` verifies a no-op `set_memory_score` call after extractor write produces identical second-call results.
4. **Dead code removed** (C2) — folded into the lock-restructure; the unused `tmp_path` is gone.
5. **Replace-all duplicates + CRLF tolerance** (X3 / C5) — `_splice_memory_score_frontmatter` now uses `count=0` to remove ALL existing importance/salience lines (handles malformed frontmatter with duplicates) and `.*?\r?` to absorb optional CR (CRLF tolerance). Then appends one canonical line for each scalar at the end of the frontmatter block.
6. **`FileBus.subscribe` malformed-row guard** (X4) — `ContextItem` construction wrapped in `try/except (TypeError, ValueError): continue`. A row with a string-typed `importance` is now skipped, mirroring the existing malformed-JSON skip path.
7. **Parent-dir fsync deferred** (X5) — out of scope; documented in this report. The current implementation uses `os.replace`, which is atomic on POSIX. Durable rename across power loss requires `fsync(parent_dir_fd)` and is a separate hardening pass.

### Tests added

- `test_set_memory_score_concurrent_partial_updates_serialize` — Codex's lost-write scenario. Two threads, one updating `importance` only, one updating `salience` only. Asserts BOTH updates land (no silent overwrite). Without the file lock, this test would fail intermittently.
- `test_set_memory_score_rejects_string_typed_input` — passes `importance: "0.88"` (string), asserts `{ok: false, error: "...number..."}`.
- `test_extractor_emits_byte_equivalent_format_to_splice` — writes via decision_extractor, calls set_memory_score with same values, asserts no-op idempotency at the second call.
