2. Harness orchestration

2.1 Model routing policy (cost ⇄ quality balance)

Work classDev modelReview modelRationale
Security-critical (auth, ACL filters, export gating, crypto) CL-ODS + GM (dual review) Highest-stakes code gets the strongest dev model and two independent reviewers.
Core backend (services, parsers, sync engine, connectors) CL-SDS Sonnet quality at sustainable cost; DeepSeek is a genuinely different model family for review.
Frontend (React/TS components, styling, state) CL-SGM Gemini reviews UI code well and its long context handles whole-component-tree review cheaply.
Tests, fixtures, docstrings, scaffolding, lint sweeps OL or CL-HCL-S (spot-check) Cheap models for high-volume low-risk output; Sonnet spot-checks 1-in-N.
Large-file / whole-module audits (>50K tokens) —GM Gemini's context window audits entire modules in one pass.
Consensus tiebreak (advisory only) —OR Third independent opinion before escalating to you; never decides alone.

2.2 Consensus protocol (your dev/review standard, codified)

1. DEV    : dev-model implements ticket on its lane branch; self-tests pass.
2. REVIEW : review-model reviews the diff with a structured checklist
            (correctness, security, perf, conventions, test coverage).
3. AGREE? : reviewer approves → merge to lane branch → ticket done.
4. REBUT  : reviewer objects → dev-model gets ONE rebuttal round
            (fix or written defense).
5. STILL SPLIT → OR (OpenRouter) renders an advisory third opinion.
6. ESCALATE: if 2-of-3 consensus is still absent, a DECISION ticket is
            written to docs/decisions/V2-DEC-NNN.md containing:
            - the question, each model's position, pros/cons table,
              recommendation strength, and blast radius.
            → surfaced to YOU. Work on that ticket pauses; the lane
              continues with the next unblocked ticket.
All decision records live in docs/decisions/ with the V2-DEC- prefix so they're auditable after merge. The harness never silently picks a side on a split decision.
