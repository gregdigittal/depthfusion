DepthFusion runs one central memory server on your VPS. Every Claude Code session — your Mac, a teammate's laptop, a CI server, any account — reads from and writes to the same knowledge base automatically. Part A
VPS Setup — Done Once by the Admin
Estimated time: 15–20 minutes. You need SSH access to your VPS.
A1
Prerequisites and Install on the VPS
SSH into your VPS and run these checks first
VPS terminal — check prerequisites
$ python3 --version    # Must be 3.10, 3.11, or 3.12
$ git --version         # Must be installed
⚠️
Python 3.10+ required
Ubuntu 20.04 ships Python 3.8. If that's what you have:
sudo apt install python3.11 python3.11-venv python3-pip

Clone the repository and install:

VPS — clone, create venv, install
$ cd ~/projects
$ git clone https://github.com/gregdigittal/depthfusion.git
$ cd depthfusion
$ python3.11 -m venv .venv
$ source .venv/bin/activate
(.venv) $ pip install -e .
(.venv) $ pip install -e ".[vps-cpu]"   # fastapi + uvicorn for the HTTP server
ℹ️
Already installed?
If DepthFusion is already on your VPS, just make sure the vps-cpu extras are installed: source ~/projects/depthfusion/.venv/bin/activate && pip install -e ".[vps-cpu]"

Run the installer to write the config file and register hooks:

VPS — run the installer
(.venv) $ python3 -m depthfusion.install.install --mode vps-cpu

[INSTALL] Configuring DepthFusion for VPS-CPU mode
[INSTALL]   Wrote config: /home/gregmorris/.claude/depthfusion.env
[INSTALL]   Registered PreCompact hook
[INSTALL]   Registered PostCompact hook
[INSTALL] VPS-CPU install complete.
A2
Generate the Bearer Token
This is the password every developer will use to connect. Keep it secret.
VPS — generate a cryptographically secure token
$ openssl rand -hex 32
a3f9c2b1d8e7f4a2c6b9d3e1f8a2b4c7d9e3f1a2b4c6d8e1f3a5b7c9d2e4f6
# ↑ Copy this. You will need it in the next step AND to share with developers.
🛑
Anyone with this token can read and write your team's memory
Share it only over a secure channel (encrypted message, password manager, etc.). Do not put it in Slack, email, or any git repository.
A3
Configure the Server
Edit the env file to enable the public HTTP server with your token
VPS — open the config file
$ nano ~/.claude/depthfusion.env

The file already exists from the installer. Add or uncomment these four lines at the bottom:

~/.claude/depthfusion.env — add these lines
# HTTP/SSE transport — enables remote access for all machines
DEPTHFUSION_MCP_HTTP_ENABLED=true
DEPTHFUSION_MCP_PUBLIC=1
DEPTHFUSION_MCP_PORT=7301
DEPTHFUSION_MCP_TOKEN=a3f9c2b1d8e7f4a2c6b9d3e1f8a2b4c7d9e3f1a2b4c6d8e1f3a5b7c9d2e4f6
# ↑ Replace with your actual token from Step A2
⚠️
DEPTHFUSION_MCP_TOKEN is mandatory when DEPTHFUSION_MCP_PUBLIC=1
The server refuses to start if DEPTHFUSION_MCP_PUBLIC=1 is set without a token. This is a safety guard — the token must be set before the server will bind to a public interface.

Save and close the file (Ctrl+O, Enter, Ctrl+X in nano).

A4
Start the HTTP Server as a systemd Service
This keeps the server running 24/7, surviving reboots and crashes automatically
VPS — create the systemd unit file
$ sudo nano /etc/systemd/system/depthfusion-mcp.service

Paste this exactly, replacing gregmorris with your VPS username:

/etc/systemd/system/depthfusion-mcp.service
[Unit]
Description=DepthFusion MCP HTTP Server
After=network.target

[Service]
User=gregmorris
WorkingDirectory=/home/gregmorris/projects/depthfusion
EnvironmentFile=/home/gregmorris/.claude/depthfusion.env
ExecStart=/home/gregmorris/projects/depthfusion/.venv/bin/python3 \
    -m depthfusion.mcp.http_server
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target

Save, then enable and start the service:

VPS — enable and start
$ sudo systemctl daemon-reload
$ sudo systemctl enable depthfusion-mcp
$ sudo systemctl start depthfusion-mcp

# Confirm it is running:
$ sudo systemctl status depthfusion-mcp
● depthfusion-mcp.service - DepthFusion MCP HTTP Server
   Active: active (running) since ...   ← must say this
ℹ️
Checking logs if something goes wrong
sudo journalctl -u depthfusion-mcp -f shows live logs. The most common startup error is a missing or commented-out DEPTHFUSION_MCP_TOKEN — check the env file if you see ValueError.
A5
Open the Firewall Port
Port 7301 must be reachable from each developer's machine

Run the command that matches your VPS setup:

VPS — open port 7301 (pick one)
# Option 1: ufw (Ubuntu/Debian default)
$ sudo ufw allow 7301/tcp

# Option 2: iptables
$ sudo iptables -A INPUT -p tcp --dport 7301 -j ACCEPT

# Option 3: Cloud provider security group
# → AWS: EC2 → Security Groups → Inbound Rules → Add TCP 7301
# → Hetzner: Firewall → Add Rule → TCP 7301
# → DigitalOcean: Networking → Firewalls → Add Rule → TCP 7301
⚠️
The bearer token is your security — the port can be open
Port 7301 is protected by the bearer token. No valid token = no access to memory. That said, for extra security you can restrict the inbound rule to specific IP ranges (your office network, developer home IPs) instead of 0.0.0.0/0.
A6
Test the Server Is Reachable
Run this from your local machine (not the VPS) to confirm it's publicly accessible
Your local machine — test from outside the VPS
# Health check — no token required
$ curl http://YOUR_VPS_IP:7301/health
{"status":"ok","transport":"sse","version":"1.0.0"}   ← success

# Authenticated check — must return 200, not 401
$ curl -H "Authorization: Bearer YOUR_TOKEN" \
     http://YOUR_VPS_IP:7301/sse
event: endpoint
data: /messages?sessionId=...   ← success (Ctrl+C to stop)
🛑
curl: Connection refused?
Either the firewall is still blocking port 7301, or the service isn't running. Check:
1. sudo systemctl status depthfusion-mcp on the VPS
2. sudo ufw status on the VPS — confirm 7301 is listed
3. Your cloud provider's security group / firewall rules in the web console
✅
VPS setup is complete
The server is running and reachable. You can now onboard developers — share the VPS IP address and the bearer token with each person.

Part B
Developer Onboarding — One Command Per Machine
Each developer runs this on their own machine. Takes under 2 minutes. No local install of DepthFusion needed.
B1
Connect Claude Code to the VPS
Run this on every machine and every Claude account you want to connect
ℹ️
What you need from the admin
Before running this, get two things from the admin:
1. The VPS IP address or hostname (e.g. 176.9.147.206 or memory.yourcompany.com)
2. The bearer token
⚠️
Upgrading from a previous install?
If you already have a depthfusion MCP registration (local or SSE), remove it first:
claude mcp remove depthfusion -s user
Then run the command below. The --header flag is not supported by this version of the Claude CLI — use the URL only; the bearer token is for curl/API verification (see Step A6), not for CLI registration.
Developer's machine — register the shared memory server
$ claude mcp add depthfusion \
    --transport sse \
    http://YOUR_VPS_IP:7301/sse

# Confirm it registered:
$ claude mcp list
depthfusion   ← must appear here

Then restart Claude Code (quit and reopen, or start a fresh claude session in the terminal). The connection is established at session start.

ℹ️
Multiple accounts on the same machine
Run the same claude mcp add command for each Claude account. The --scope user flag (added automatically) ties the registration to the current user profile. Each account that runs this command gains access to the shared memory.
ℹ️
Already registered a local (stdio) DepthFusion?
Remove it first and replace with the remote endpoint:
claude mcp remove depthfusion
Then re-run the claude mcp add command above.
B2
Verify the Connection Works
Run these checks in order — each confirms a different layer
🔬 Check 1 — MCP is listed
$ claude mcp list
depthfusion   ← must appear
🔬 Check 2 — Health endpoint reachable from this machine
$ curl http://YOUR_VPS_IP:7301/health
{"status":"ok","transport":"sse","version":"1.0.0"}
🔬 Check 3 — Live MCP tool call from Claude Code

Open a new Claude Code session and type:

Can you call depthfusion_status and show me the output?
✅
Successful response
Claude replies with component status, active mode, bus backend, graph node/edge counts, and engaged features. This confirms the full MCP round-trip is working.
🛑
Claude says the tool is unavailable?
The MCP server didn't connect. Go back to Step B1 — check the URL has no typo, the token matches exactly, and that you restarted Claude Code after registering.
B3
First Recall Test
Confirm that past sessions are actually being retrieved

In any Claude Code session, ask:

Call depthfusion_recall_relevant with query "recent work" and show me the blocks returned.
ℹ️
Getting 0 blocks on a fresh install?
That is normal. The memory store builds up as sessions are captured. The first blocks appear after the first context compaction (/compact in Claude Code) or after the auto-compress job runs. After a few sessions you'll see relevant context loading automatically at every session start.

Part C
What Syncs Automatically
Nothing extra to configure — this happens once the server is running and all machines are connected.
C1
Automatic Data Flow
Every session on every connected machine contributes to and benefits from the shared store
  • 🚀
    Session start: DepthFusion queries the VPS for the most relevant past context and injects it before Claude responds to your first message. This works on every machine — no action needed from the developer.
  • 💾
    Pre-compact: Before Claude Code compresses its context window, a snapshot of the active session is saved locally on the VPS at ~/.claude/.depthfusion-compact-snapshot.json
  • 🧠
    Post-compact: After each compaction, the session transcript is analysed. Significant decisions, code patterns, conventions, and preferences are extracted and stored as discovery files at ~/.claude/shared/discoveries/ on the VPS — where every connected machine can recall them.
  • 📊
    Knowledge graph: When DEPTHFUSION_GRAPH_ENABLED=true, entities (projects, tools, decisions, people) and their relationships are extracted and stored in a graph database. The depthfusion_graph_traverse tool lets you explore connections across the team's entire work history.
  • ⏳
    Memory decay: Stale memories fade automatically. High-importance items lose 1%/day of salience. Low-importance items lose 5%/day. Items that drop below 0.05 salience move to an archive folder — never deleted, always recoverable. This keeps the memory store lean and relevant.
  • 🔁
    Recall feedback: When Claude retrieves context and uses it, calling depthfusion_recall_feedback increases the salience of useful chunks. Things the team keeps finding valuable rise to the top over time.
✅
Cross-machine flow in practice
Developer A decides on an architecture approach in a Monday session on their MacBook. That decision is captured in the VPS after their first compaction. On Tuesday, Developer B opens a Claude session on their laptop about the same project — DepthFusion automatically injects the relevant context from Developer A's session at the start of the conversation.
C2
Project-Scoped vs Global Recall
Control whether recall searches one project or all projects across the whole team
ScopeWhat it searchesHow to set it
project Only sessions and discoveries tagged with the current project slug. Default. Automatically detected from the git repo name in the current directory.
cross_project All projects this developer has worked on. Tell Claude: "Call depthfusion_set_scope with scope=cross_project"
global The entire team's memory across every project ever captured. Tell Claude: "Call depthfusion_set_scope with scope=global"
💡
Recommended for team use
Start new sessions with cross_project scope. It surfaces architectural patterns and conventions from across the codebase, not just the one file you're editing. Switch to project when you need focused, noise-free recall.

Part E — Optional · Apple Silicon Macs only
Local GPU Inference via MLX
Run Gemma or Qwen on your Mac's Metal GPU for reranking, extraction, and summarisation — no API calls, no latency, no cost. Works alongside the VPS hub.
💡
What mac-mlx adds on top of the base VPS setup
Without mac-mlx: reranking is done by Haiku via the Anthropic API (or skipped in local mode). With mac-mlx: a local Qwen or Gemma model runs on your Mac's Metal GPU. DepthFusion sends reranking and extraction tasks to this model — instant responses, fully private, and free. Shared memory still flows to and from the VPS hub unchanged.
E1
Prerequisites and Install
Apple Silicon (M1/M2/M3/M4) required — Intel Macs are not supported by mlx_lm
⚠️
Apple Silicon only
mac-mlx mode requires an arm64 macOS machine. Run uname -m — it must return arm64. The installer refuses on Intel Macs and non-macOS systems. Use --mode vps-cpu on Intel Macs instead.
⚠️
Upgrading from a previous install?
If you already have a depthfusion MCP registration, remove it before re-registering:
claude mcp remove depthfusion -s user
Also run git pull origin main before pip install if your clone predates the mac-mlx commit — an older clone will warn "does not provide the extra 'mac-mlx'" and the installer will reject --mode mac-mlx.

Clone or update the repository on your Mac, then install the mac-mlx extras:

Mac terminal — install mac-mlx extras
# Navigate to your local depthfusion clone
$ cd ~/projects/depthfusion

# Pull latest (required if clone predates the mac-mlx commit)
$ git pull origin main

# Create or activate your venv
$ python3 -m venv .venv
$ source .venv/bin/activate

# Install mac-mlx extras (mlx-lm + sentence-transformers + fastapi)
(.venv) $ pip install -e ".[mac-mlx]"

# Confirm mlx_lm is available
(.venv) $ python3 -c "import mlx_lm; print('mlx_lm OK')"
mlx_lm OK
ℹ️
Already have a local DepthFusion install?
Just re-run pip inside your existing venv: pip install -e ".[mac-mlx]". No other changes needed — the extras add mlx-lm and sentence-transformers without removing anything.

Register DepthFusion with Claude Code using the wrapper script (avoids the -m flag parsing issue):

Mac terminal — register MCP server (stdio, local mode)
# If re-registering: remove the old entry first
$ claude mcp remove depthfusion -s user

# Register using the wrapper script (claude CLI intercepts -m, so use the script)
$ claude mcp add depthfusion -s user ~/projects/depthfusion/scripts/mcp-server.sh

$ claude mcp list
depthfusion   ← must appear here
ℹ️
Why the wrapper script?
Running claude mcp add depthfusion -s user python3 -m depthfusion.mcp.server fails with error: unknown option '-m' because the Claude CLI intercepts the -m flag before passing arguments to the subprocess. scripts/mcp-server.sh wraps the call with exec python3 -m depthfusion.mcp.server, bypassing this.
⚠️
If you also want team shared memory (recommended)
After running the mac-mlx installer, follow Part B to register the VPS as your MCP server. The mac-mlx mode provides local GPU inference; the VPS provides shared team memory. They are independent layers that complement each other — the Mac's model handles reranking quality while team discoveries flow to and from the VPS hub.

Need more headroom on 36 GB?
Qwen2.5-32B uses ~20 GB, leaving only ~16 GB for macOS and other running apps — which can be tight if you run browsers, Docker, or other memory-hungry tools simultaneously. For a more comfortable experience on 36 GB, choose [2] Qwen2.5-14B (~9 GB, leaves ~27 GB free). Quality is slightly lower than 32B but still well above the Haiku API fallback.
E4
Using mac-mlx Alongside the VPS Hub
Local GPU intelligence + shared team memory — two independent layers that compose cleanly
  • 🔥
    Local inference layer (mac-mlx): When mlx_lm.server is running on port 8000, DepthFusion routes reranking, extraction, summarisation, and linking to your Mac's GPU via GemmaBackend. Results are instant, private, and cost nothing. If the server is not running, GemmaBackend is unhealthy and the factory chain falls back to HaikuBackend (Anthropic API) automatically — no manual intervention needed.
  • 🌐
    Shared memory layer (VPS hub): The MCP server registered in Part B continues to read and write discoveries to the VPS hub over SSE. Teammates' sessions remain visible in recall. The mac-mlx reranker improves how those shared memories are scored and returned — it does not change where they live or who can access them.
  • 🔄
    Fallback chain: The factory quality chain for mac-mlx mode is gemma → haiku → null. GemmaBackend (local MLX model) is tried first; if unhealthy, HaikuBackend (API) is tried next; if that is also unavailable, NullBackend returns empty results gracefully. Your session never crashes.
  • 💡
    Solo use without a VPS: mac-mlx works without the VPS hub. Memory is read from and written to your local ~/.claude/shared/discoveries/ only. Local GPU inference still works fully — you just don't share memory with teammates. Useful for solo developers or air-gapped environments.
✅
Full stack on a 36 GB Mac
With DEPTHFUSION_MODE=mac-mlx and Qwen2.5-32B running on port 8000: reranking latency drops to under 500 ms, extraction runs locally with no API cost, and all results still sync to the VPS hub for team recall. You get both layers — local GPU speed and shared team memory.
