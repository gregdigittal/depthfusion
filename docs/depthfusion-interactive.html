How DepthFusion works end-to-end

From session end to future recall — the full data lifecycle.

1

Session ends → hooks fire automatically

Claude Code's PreCompact and PostCompact lifecycle hooks trigger DepthFusion. The pre-compact hook snapshots current state; the post-compact hook extracts knowledge using heuristic regex or optional Haiku summarization.

# ~/.claude-shared/hooks/depthfusion-post-compact.sh capture: decisions, warnings, facts → ~/.claude/shared/discoveries/ extract: entities, relationships → ~/.claude/depthfusion-graph.json
2

Knowledge stored as discovery files with frontmatter

Each extracted fact becomes a markdown file with YAML frontmatter encoding metadata. These files persist across all future sessions.

--- project: depthfusion importance: 0.9 salience: 2.4 tags: [infra, redis, security] pinned: false --- Redis must bind to 127.0.0.1:6379. requirepass is mandatory...
3

New session → recall query issued via MCP

DepthFusion's session-start hook injects project context. When you or Claude issues a recall query, the MCP server executes the retrieval pipeline.

# MCP tool call depthfusion_recall_relevant({ "query": "Redis config and port binding", "top_k": 5, "project": "depthfusion" })
4

BM25 scoring with source weighting

Every block from sessions, discoveries, and memory files is scored using BM25 (k1=1.5, b=0.75). Scores are multiplied by source trust weights — user-written memory is most reliable, machine-generated sessions are least.

source weights: memory × 1.00 # user-written, most reliable discovery × 0.85 # extracted facts session × 0.70 # machine-generated
5

Deduplication → optional semantic reranking

Only the highest-scoring chunk per file is kept before reranking — preventing redundant content. On VPS tiers, Haiku evaluates the top candidates for semantic relevance beyond keyword overlap.

6

Top-K results injected into Claude's context

Only the most relevant blocks enter context — not your entire history. Each result carries a recall_id enabling the feedback loop.

# Response { "blocks": [ ... 5 results ], "recall_id": "uuid-for-feedback", "total_sources_scanned": 70, "engaged_layers": ["bm25", "reranker"] }
7

Feedback loop adjusts salience scores

When you confirm which results were useful, salience updates. Used blocks become more likely to surface again; ignored blocks gently decay. Retrieval improves with every session.

depthfusion_recall_feedback({ "recall_id": "uuid", "used": ["redis-config.md"], # salience += 0.1 "ignored": ["old-session.tmp"] # salience -= 0.05 })

Token Efficiency

Naive approaches flood your context with irrelevant history. DepthFusion retrieves only what's relevant — measured per-query.

Naive approach — dump all context
Full MEMORY.md file~4,200 tok
Recent session dumps (3×)~6,800 tok
All discovery files~3,100 tok
Context used per query ~14,100 tokens
DepthFusion — query-aware retrieval
Relevant memory blocks (top-3)~680 tok
Relevant discovery snippets (2)~420 tok
Session context (session-init)~340 tok
Context used per query ~1,440 tokens
Typical token reduction per recall query: ~90% fewer tokens

Block-level chunking

Files are split on ## headers. A 50KB BACKLOG.md with 20 sections → 20 independently-scored blocks. Only the highest-scoring section enters context.

Deduplication gate

Before reranking, only the top-scoring chunk per file is kept. The Haiku reranker never sees semantically similar content twice — reducing API cost by up to 90%.

Salience-based pruning

Old discoveries with low usage scores decay gradually and archive automatically at salience < 0.05. BM25 runs on a leaner, higher-quality corpus over time.

Snippet truncation

Results are truncated at configurable snippet_len (default 1,500 chars), cut at sentence boundaries. Signal without padding token waste.

Salience, Decay & the Feedback Loop

DepthFusion's memory isn't static — it learns which knowledge matters. Salience scores shift with every recall, making future retrieval more accurate.

What's New in v0.6.0

First stable release of the 0.6 series. Two P0 correctness fixes make advertised modes functional for the first time, plus the full E-30 build-plan alignment (8 stories, +89 tests).

🔴 P0 Fixes — Upgrade Immediately If on vps-cpu or vps-gpu

Both bugs caused silent degradation to BM25-only retrieval with no error message.

S-86 · Mode Resolution

DEPTHFUSION_MODE=vps-cpu silently fell through to local BM25-only retrieval. utils/mode.py::normalise_mode() now correctly routes to the Haiku reranker chain.

S-87 · Missing Anthropic SDK

pip install depthfusion[vps-cpu] did not install anthropic>=0.40. The Haiku reranker silently degraded to NullBackend on every clean install. Fixed in pyproject.toml.

S-88 · MCP Tool Schemas

All 18 MCP tools now return typed JSON schemas with required arrays, documented parameter bounds, and enum constraints. Enables type-safe client generation.

S-89 · Vector Consistency

ChromaDB add_document() and query() both explicitly call get_backend("embedding"), eliminating silent vector-space mismatch between index and query embeddings.

S-90 · Benchmark Harness

scripts/benchmark.py + tests/fixtures/recall_goldset.jsonl (8 entries). Measure recall precision/MRR with no API key required. Portable across all three modes.

S-91 · SQLite Metadata Cache

storage/file_index.py::FileMetadataIndex — WAL-mode SQLite cache skips unchanged files on warm recall. Reduces re-hashing overhead on large project corpora.

S-92 · Recall Explainability

explain: true on depthfusion_recall_relevant returns bm25_score, rrf_score, source_weight, and reranker_rank per result block.

S-35 AC-4 · RLM HTTP Sidecar

FastAPI sidecar at 127.0.0.1:8771 (loopback-only). Exposes GET /health, POST /run, GET /schema for recursive learning module calls.
