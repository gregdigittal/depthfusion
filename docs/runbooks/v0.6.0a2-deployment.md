## Deployment procedure (mid-session safe)

Pick the procedure matching your install mode.

### `local` mode (laptop, no API)

```bash
cd ~/projects/depthfusion
git fetch --tags
git checkout v0.6.0a2

# No pip install needed — no new dependencies in v0.6.0a2.
# Existing Claude Code sessions keep running on v0.6.0a1 code (in-memory).
# New sessions started after this point will load v0.6.0a2.
```

That's it. Nothing else to do.

### `vps-cpu` mode (cloud server with Haiku API)

SSH to the VPS, then:

```bash
cd ~/projects/depthfusion
git fetch --tags
git checkout v0.6.0a2

# No pip install needed — pyproject.toml dependency list is unchanged in v0.6.0a2.
# (If you HAVE local changes to pyproject.toml or want to verify dep consistency:
#   pip install -e '.[vps-cpu]' --no-deps
# Skip this if you just pulled — it's a no-op.)

# No service restart needed. The MCP server is a per-session subprocess;
# new sessions pick up the new code automatically.
```

### `vps-gpu` mode (CUDA host with local Gemma)

SSH to the GPU host (e.g. hetzner-gpu), then:

```bash
cd ~/projects/depthfusion
git fetch --tags
git checkout v0.6.0a2

# Same as vps-cpu — no pip install, no service restart needed.

# IMPORTANT: do NOT restart the vLLM systemd service.
# v0.6.0a2 makes no changes to vLLM config, model paths, or the Gemma backend.
# Restarting vLLM would interrupt in-flight inference for active sessions.
sudo systemctl status vllm-gemma   # confirm it's still running on the same model
```

## Post-deployment verification (~2 minutes)

These checks confirm the deploy worked. Run from the deploying machine after at least one new Claude Code session has started post-checkout.

### 1. Confirm the new code is loaded in fresh sessions

Start a fresh Claude Code session, then in another terminal:

```bash
TODAY=$(date -u +%Y-%m-%d)
SIMPLE="$HOME/.claude/depthfusion-metrics/$TODAY.jsonl"

# system.startup event should appear (S-79 AC-3 — new in v0.6.0a2)
grep -c '"metric":"system.startup"' "$SIMPLE" 2>/dev/null
# Expected: ≥ 1 (one per new MCP server init since deploy)

# Confirm the server_version label
grep '"metric":"system.startup"' "$SIMPLE" 2>/dev/null | tail -1 | python3 -c "import sys,json; print(json.loads(sys.stdin.read())['labels']['server_version'])"
# Expected: 0.6.0a2
```

### 2. Confirm new fields populate in recall events

After triggering a recall (any session-start auto-recall counts), inspect today's recall stream:

```bash
RECALL="$HOME/.claude/depthfusion-metrics/$TODAY-recall.jsonl"

# Check the latest event has all six per-capability latencies (S-80)
tail -1 "$RECALL" | python3 -c "
import sys,json
e = json.loads(sys.stdin.read())
keys = set(e.get('latency_ms_per_capability', {}).keys())
expected = {'extractor','linker','summariser','embedding','decision_extractor','reranker'}
missing = expected - keys
print('S-80 latency keys:', 'OK' if not missing else f'MISSING {missing}')
print('S-81 config_version_id:', e.get('config_version_id', '<absent>'))
print('S-83 backend_fallback_chain:', e.get('backend_fallback_chain', '<absent>'))
"
# Expected:
#   S-80 latency keys: OK
#   S-81 config_version_id: <12-char hex string>  (NOT empty, NOT "none")
#   S-83 backend_fallback_chain: {capability: [...], ...}  (NOT empty {})
```

If any of these fields are missing or empty after a fresh session, the new code is not loading. Check that the venv path in `~/.claude-shared/hooks/depthfusion-session-init.sh:11` resolves to your updated checkout.

### 3. Confirm no regressions

```bash
# Old fields still present (S-80 didn't break the existing reranker timing)
tail -1 "$RECALL" | python3 -c "import sys,json; e=json.loads(sys.stdin.read()); print('total_latency_ms:', e.get('total_latency_ms'))"
# Expected: numeric value, not None

# Backend fallback events still write to legacy stream (S-83 kept these as complementary)
grep -c '"metric":"backend.runtime_fallback"\|"metric":"backend.fallback"' "$SIMPLE" 2>/dev/null
# Expected: ≥ 0 (only fires on actual fallbacks; absence is fine if no fallbacks happened today)
```

## Rollback procedure

If anything looks wrong after deploy:

```bash
cd ~/projects/depthfusion
git checkout v0.6.0a1                    # roll back to the previous tag

# No further action needed. Currently-running sessions are unaffected
# (they still hold their original code in memory). New sessions will
# load v0.6.0a1 source.
```

Rollback is non-destructive because:
- No data migrations occurred
- No file format changes (JSONL streams are append-only and forward-compatible)
- No installed-dependency changes
- The `system.startup` events written under v0.6.0a2 remain in the JSONL streams (harmless to v0.6.0a1 readers)
