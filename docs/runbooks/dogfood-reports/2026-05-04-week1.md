## Headline finding

**Zero production-path capture events were emitted in 13 days.** All 957 capture
events recorded under `~/.claude/depthfusion-metrics/` write to `/tmp/...`
paths (test fixtures and harness runs). All 30 recall events cluster into 7
minute-buckets across 4 dates (9-event bursts characteristic of test suites,
not organic distribution). The instrumentation works in tests; it has not been
exercised by a real Claude Code session over the full observation window.

Either (a) Claude Code sessions on this host aren't invoking the depthfusion
MCP server, (b) the MCP server is invoked but the capture/recall emitters
aren't reached, or (c) emissions are being routed to a different metrics
directory. Root-cause investigation is filed as **S-79 (P0)** below.

This is the most consequential dogfood signal: the v0.5.1 / v0.5.2
observability layer is unvalidated against real workloads.

## Recall-stream findings

30 events recorded across 7 distinct minute-buckets. Modes seen: `vps`,
`vps-cpu`, `vps-gpu`. All `event_subtype: "ok"`.

**Field-by-field issues:**

- `latency_ms_per_capability` — populated **only for `reranker`**, and only on
  10 / 30 events. The other 20 events have an empty `latency_ms_per_capability:
  {}`. Five other capabilities (`extractor`, `linker`, `summariser`,
  `embedding`, `decision_extractor`) appear in `backend_used` on every event
  but never appear in the latency dict. **S-43 AC-3 (p95 recall latency
  ≤ 1500 ms with 100-file corpus) cannot close from this data** — the
  per-capability breakdown the AC implies is structurally absent.

- `backend_fallback_chain` — empty `{}` on **all 30 events**. The legacy stream
  for the same dates contains 981 fallback events (505 `backend.fallback` +
  476 `backend.runtime_fallback`). Two emission paths for the same data; only
  the legacy path produced anything. The structured field is dead. (Earlier
  drafts of this report quoted 931 / 483 / 448 — corrected 2026-05-05 after
  independent review re-tallied the JSONL.)

- `config_version_id` — empty string `""` on **all 30 events**. Per the
  inline comment at `src/depthfusion/retrieval/hybrid.py:188-191`, this field
  is mandatory under DR-018 §4 (D-3 invariant — auditors can reproduce any
  gate decision against the exact GateConfig that produced it). It is
  structurally present but always blank in real emissions. The invariant is
  unenforced.

**What works:** `backend_used` is consistently populated with all 6
capabilities. `total_latency_ms` is always present. `query_hash` truncates to
12 chars correctly.

The mechanism mix is plausibly proportional to a healthy capture pipeline —
but every single event is from a test fixture (file paths under `/tmp/...`,
project names like `myproj` / `myapp` / `testproj` / `p1`). No event traces
back to a real `~/.claude/depthfusion-discoveries/{project}/` write.

**Field-by-field issues:**

- `config_version_id` — empty string `""` on **all 957 events**. Same

  threaded through `MetricsCollector` at all in non-gate code paths.

- `entries_written` — populated (1 or 2 in samples) and looks correct.

- `write_success` — always `true` in the sample. No failure cases observed.
  Either the capture pipeline is genuinely robust or the failure path is
  never exercised in tests. Cannot distinguish without real-session data.

**What works:** `event`, `event_subtype`, `capture_mechanism`,
`capture_mechanism_known`, `project`, `session_id`, `write_success`,
`entries_written`, `file_path`, `timestamp` are all populated. The schema is
sound; the invocation surface is the problem.

## Gates-stream findings

**No gates events recorded in 13 days.** Root cause is not a wiring bug:
`record_gate_log()` is correctly called from
`src/depthfusion/retrieval/hybrid.py:196` inside `apply_fusion_gates`. But
that method short-circuits at line 161 unless
`DEPTHFUSION_FUSION_GATES_ENABLED` is set:

```python
if os.environ.get("DEPTHFUSION_FUSION_GATES_ENABLED", "false").lower() not in (
    "true", "1", "yes",
):
    return blocks
```

The default is `"false"` by design (see CHANGELOG.md L677, build plan
`docs/plans/v0.5/02-build-plan.md` L366, `docs/runbooks/tier-feature-matrix.md`
L25 — all four sources agree: gates are an opt-in tier feature).

**The runbook contradicts itself on this point:**

- §1 lists `-gates.jsonl` alongside the other three streams as something
  `MetricsCollector` writes, with no caveat.
- §2 says "**No env flags need setting.** All four streams emit by default
  as of v0.5.2."
- §4d (analysis checklist) admits "Should emit when
  `DEPTHFUSION_FUSION_GATES_ENABLED` is on."

§2 is wrong. The operator (me) read it, didn't set the flag, ran no gates
queries, and found an empty stream. This is a runbook bug, not a code bug.

## Cross-stream consistency

- **Fallback double-emission:** legacy stream wrote 981 `backend.fallback*`
  events (505 + 476); structured recall stream wrote 0 populated `backend_fallback_chain`
  fields. Same underlying event, two channels, only the legacy one produces
  data. Either kill the legacy path, populate the new one, or document them
  as complementary (e.g., legacy = aggregate count, structured = per-query
  detail).

- **`config_version_id` empty in 100% of events across two streams** (957 +
  30 = 987/987). This isn't an emitter bug — it's a *plumbing* bug. The hash
  isn't reaching `MetricsCollector.record_capture_event()` or
  `record_recall_query()`. The only call site that *does* compute and pass it
  is `record_gate_log()` (hybrid.py:198) — which never executed.

- **Test-fixture pollution:** `~/.claude/depthfusion-metrics/` is the
  documented production telemetry directory but currently contains 100% test
  data. pytest invocations of `MetricsCollector()` (default constructor with
  no `tmp_path`) write to the user-home directory. The tests should route to
  `tmp_path`; or the production path should be guarded against test
  invocation.

## What surprised me

1. **The "calendar-blocked ≥ 7 days" annotation on T-205 captured the wrong
   constraint.** Calendar time was a proxy for "enough co-emitted real-session
   data." After 13 days, the proxy elapsed but the underlying condition never
   engaged — because the underlying condition wasn't "time passes," it was
   "DepthFusion gets used by real sessions." Future calendar-blocked notes
   should specify the data threshold (e.g., "≥ 5 days each with all four
   streams co-emitting from at least one production-path session"), not just
   the day count.

2. **The most striking finding wasn't any individual field — it was the
   `/tmp/` prefix on every file path.** A single `ls` and a one-line Python
   tally surfaced more about the v0.5.x observability layer's state than any
   of the per-field analyses. Cheap categorical pivots beat exhaustive field
   audits when the signal is structural.

3. **The runbook's §1 → §2 → §4d contradiction would have stayed invisible
   without the dogfood pass.** Internal consistency of a 252-line runbook is
   the kind of thing only an operator running the procedure end-to-end can
   catch. Add a "self-consistency check" pass to runbook reviews going forward.

4. **`backend_used` has the schema we need; `latency_ms_per_capability`
   doesn't use it.** They're parallel structures in the same JSON object.
   The fix for S-80 is probably one line per capability — record the latency
   when the backend call returns, not just when the *reranker* returns.

5. **The legacy `*.jsonl` stream is doing more work than the
   `*-recall.jsonl` stream.** 981 useful fallback events vs 0 in the
   "structured" successor. Migration looks <50% complete. Worth a deeper look
   before declaring v0.6 ready to drop the legacy path.

## What to change in the runbook

For the next pass — these are corrections to `dogfood-telemetry.md`:

   real-session activity, check that `~/.claude/depthfusion-metrics/<today>-capture.jsonl`
   contains at least one event with a `file_path` outside `/tmp/`. If all
   paths are `/tmp/...`, only test fixtures are emitting — investigate before
   continuing the week."

3. **§7 report template:** add a "Headline finding" section above "Stream
   health." The most important finding deserves the top of the document, not
   to be inferred from a stream-health table.

4. **§6 triage table:** add a sixth row: "Substrate gap" — when the
   instrumentation works but isn't being invoked. This is a different class
   from "lying field" or "wish-we-had-it" and warrants its own escalation
   path (P0 by default; blocks all other findings until resolved).

## Methodology notes (for auditability)

All findings derived from:

```bash
ls -la ~/.claude/depthfusion-metrics/
wc -l ~/.claude/depthfusion-metrics/*.jsonl
```

…plus three Python one-liners over the JSONL files (capture mechanism /
project / session distribution; recall capability / latency-key coverage;
legacy stream metric-name tally). No proprietary data was inspected; all
events are either test fixtures (already public via the test corpus) or
empty-field summaries.

The probe took ~10 minutes from "is T-205 unblocked?" to "here are 6
follow-up stories." This is the right cost/value ratio for a dogfood pass —
exhaustive per-field audits would not have surfaced findings the structural
pivots missed.
