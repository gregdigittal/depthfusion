1. What DepthFusion Is — The 30-Second Mental Model

Before any animation directive, Claude Design must understand the core insight: DepthFusion is not a vector database, not a RAG plugin, and not a chat history store. It is a Cognitive Infrastructure Layer — a shared memory fabric that sits between multiple AI agents and gives them persistent, typed, scored, and contradiction-aware memory across sessions, projects, and time.

The Problem

Claude Code forgets everything between sessions. Teams of agents working in parallel have no shared memory. Context windows fill. Architectural decisions made Tuesday are invisible Wednesday.

The Solution

DepthFusion publishes context as typed MemoryObjects (decisions, code patterns, incidents, discoveries). It recalls them via a 4-tier hybrid search pipeline that ranks by recency, relevance, and how many agents have observed each fact.

The Differentiator

Shared memory for agent teams, not just single-agent sessions. The Event Graph Fabric (Redis Streams) broadcasts each memory event so every agent in the fleet sees it in real time.

2. Architecture — Layer-by-Layer Technical Explainer

DepthFusion's retrieval engine is a 4-tier cascade. Each tier adds precision at the cost of latency. The deployment mode determines which tiers are active. All tiers compose via Reciprocal Rank Fusion (RRF) — weighted fusion outperforms plain RRF on 90% of queries (45/50 in internal benchmark).

Tier 1
BM25 — Best Match 25 Lexical Search

The baseline tier. BM25 scores documents by term frequency (TF) and inverse document frequency (IDF) with saturation curves that prevent keyword-stuffing from dominating. In DepthFusion, BM25 runs in-process on the MemoryStore (a structured JSON store backed by ~/.depthfusion/) with no network round-trip. Sub-millisecond latency even on 10,000+ stored memories. Precision@1 = 1.000 on the internal goldset (n=8 canonical test queries). Active in all deployment modes — this is the only tier available in local mode.

What BM25 is good at: exact term matches, code identifiers, project names, function names. What it misses: semantic synonyms ("auth" vs "authentication"), conceptual similarity.

Tier 2a
Haiku LLM Reranker (vps-cpu mode)

A Claude Haiku API call that re-scores BM25 candidates using natural language understanding. Haiku receives the query + top-N candidates and returns confidence scores. This tier catches semantic gaps BM25 misses — e.g. a query about "session persistence" correctly surfaces a memory tagged "cross-session continuity". Cost: ~$0.0005 per rerank call (Haiku pricing). Latency: 80–200ms P50 (network-bound, not compute-bound). Can be disabled if API budget is a concern: set DEPTHFUSION_RERANKER=off.

Tier 2b
ChromaDB Vector Search (vps-cpu mode)

ChromaDB maintains a persistent embedding index of all published memories. Default embedding model: all-MiniLM-L6-v2 (via sentence-transformers, loaded only in vps-cpu/vps-gpu/mac-mlx modes). ChromaDB uses HNSW internally for its own index, but DepthFusion also exposes a standalone HNSW index (Tier 3) for higher-performance scenarios. ChromaDB is the persistent vector store; standalone HNSW is the performance index.

When vps-cpu mode is active, every depthfusion_publish_context call auto-embeds the content and upserts into ChromaDB. Recall latency: 5–25ms for collections under 50k documents.

Tier 3
HNSW Approximate Nearest Neighbor (vps-gpu / mac-mlx / hnsw extra)

Hierarchical Navigable Small World (HNSW) from hnswlib. HNSW builds a multi-layer proximity graph where high layers are long-range connections and lower layers are local. At query time, it navigates the graph greedily — O(log N) rather than the O(N) of brute-force cosine search. At 1M+ memories, HNSW maintains sub-10ms recall; ChromaDB's brute-force would take seconds. DepthFusion's HNSW index uses sentence-transformers embeddings (768-dimensional vectors for the default model). Also available as standalone via pip install depthfusion[hnsw].

mac-mlx mode: embedding inference runs via mlx_lm.server on Apple Silicon GPU, delivering ~3× throughput improvement over CPU sentence-transformers. vLLM serves the same role on CUDA GPUs in vps-gpu mode.

2.2 The fabric_seed Ranking Formula

After RRF fusion, results are re-ranked by the fabric_seed formula — a heuristic that surfaces "hot knowledge": facts that are recent, relevant, and widely observed by the agent fleet.

score = recall_relevance × recency_decay × log(1 + observer_count)
recall_relevance = normalized RRF score (0–1) · recency_decay = e−λt where t = age in days, λ = 0.1 (configurable) · observer_count = number of distinct agent principals that have retrieved this memory

recall_relevance

How well this memory matches the query. Combines BM25 rank + vector similarity + Haiku confidence score. Normalized to [0,1] before multiplication.

recency_decay

Exponential decay. A memory written today scores 1.0. A week-old memory scores ~0.5. A month-old memory scores ~0.05. This prevents stale architectural decisions from burying recent ones.

log(1 + observer_count)

Logarithmic "wisdom of the fleet" term. A memory that 10 different agents have retrieved is more likely to be load-bearing context. Log prevents any single viral fact from dominating.

Formula Animation (Act 3, ~10s): Show three dials/sliders. One represents relevance (BM25+vector bar). One represents recency (clock face unwinding). One represents observers (agent avatars accumulating). All three multiply together in real time as sliders move. Show a scenario: two equally-relevant memories, but one was observed by 8 agents vs 1 — the 8-observer one rises to the top of the ranked list despite being slightly older.

2.3 Event Graph Fabric (Redis Streams)

The Event Graph Fabric is DepthFusion's real-time fan-out layer for multi-agent teams. Enabled with pip install depthfusion[fabric] and a local Redis instance. Redis stays loopback-only (127.0.0.1:6379) — it is never exposed on external interfaces.

9 Event Types (each publish/receive creates a graph node with typed edges):
  • AGENT_PUBLISHED — agent stored a new memory
  • AGENT_RECEIVED — agent retrieved memories from recall
  • DERIVED_FROM — new memory was explicitly derived from a prior one
  • CONTRADICTS — contradiction engine fired (new fact conflicts with existing)
  • SUPERSEDES — memory explicitly marked as replacing another
  • INCIDENT_LOGGED — error/incident captured
  • DECISION_RECORDED — architectural decision captured
  • OUTCOME_REPORTED — follow-up outcome against a decision
  • SCORE_UPDATED — manual cognitive score adjustment
Graph nodes and edges are queryable via depthfusion_graph_traverse and depthfusion_graph_status.
Event Graph Animation (Act 2, ~15s): Show a canvas with three agent nodes (Agent A, Agent B, Agent C) and a central DepthFusion node. Agent A publishes — a pulsing edge appears: A→DepthFusion labeled "AGENT_PUBLISHED". 2.4 Cognitive Scoring Layer

Every stored MemoryObject is scored by the CognitiveScorer (8 components, ~15ms overhead). This is what separates DepthFusion from naive vector stores — stored facts are evaluated for their epistemic weight, not just embedded.

2.5 ContradictionEngine

The ContradictionEngine scans incoming memories against the corpus for factual conflicts. It uses a combination of semantic similarity (are these about the same topic?) and logical negation detection (do they assert opposite facts?). When a contradiction is found:

  • Both memories are stored (DepthFusion never silently discards)
  • The newer memory gets a contradiction_risk score component flagged
  • The graph records a CONTRADICTS edge between the two
  • The depthfusion_recall_relevant response surfaces the contradiction warning

2.6 Autonomic Consolidation Daemon

Enabled with DEPTHFUSION_AUTONOMIC=1. A background daemon thread (depthfusion-autonomic) wakes every N minutes (default 30, configurable via DEPTHFUSION_CONSOLIDATION_INTERVAL_MINUTES) and runs two passes:

  • Near-duplicate detection: finds memories with cosine similarity above threshold. Logs merge candidates (does not auto-merge without ACL confirmation).
  • Archive detection: finds memories that haven't been accessed in X days and have low importance scores. Logs archive candidates.

The daemon never crashes the MCP server — all exceptions are swallowed and logged. The server remains stable regardless of consolidation failures.

2.7 Materialisation Policy

Not every depthfusion_publish_context call results in a stored memory. The materialisation policy enforces a 3-gate pipeline:

Gate 1
Score ≥ threshold
→
Gate 2
Novelty cosine > 0.15
→
Gate 3
Capacity eviction check
→
Result
Stored

Gate 2 prevents near-duplicate memories from cluttering the corpus. Gate 3 ensures the store doesn't grow unbounded — least-recently-used low-importance memories are evicted when capacity is reached.

3. Deployment Modes — Technical Comparison

DepthFusion ships in 4 deployment configurations. The mode is set via DEPTHFUSION_MODE environment variable and determines which retrieval tiers activate, which packages are required, and what the performance/cost tradeoff is.

local mode

pip install depthfusion

Retrieval: BM25 only (Tier 1)
Storage: ~/.depthfusion/ JSON files
Embedding: None
Extras: None required
Network: Zero — fully offline

Best for: Individual developers, air-gapped environments, zero-config startup, privacy-sensitive work. No API keys needed.

Limitation: Semantic similarity search unavailable. Recall is exact-match and keyword-based only.

vps-cpu mode

pip install depthfusion[vps-cpu]

Retrieval: BM25 + ChromaDB vector (Tier 1+2b) + Haiku reranker (Tier 2a)
Storage: JSON + ChromaDB persistent collection
Embedding: all-MiniLM-L6-v2 via sentence-transformers
Extras: anthropic, chromadb, fastapi, uvicorn

Best for: Teams on shared Linux VPS (2–4 vCPU). HTTP sidecar serves the embedding API. Cost: Claude Haiku reranking charges (~$0.0005/call).

Performance: Recall latency 37–200ms P50 (network + Haiku RTT)

vps-gpu mode

pip install depthfusion[vps-gpu]

Retrieval: BM25 + ChromaDB + HNSW (all tiers)
Embedding: sentence-transformers on CUDA GPU (vLLM serves local LLM if needed)
Extras: sentence-transformers, chromadb, hnswlib, fastapi, uvicorn

Best for: Agent teams doing high-volume recall. No per-call API cost once GPU is provisioned. HNSW scales to millions of memories.

vLLM integration: An OpenAI-compatible local LLM (e.g. Gemma) replaces Haiku for reranking — zero marginal cost per query.

mac-mlx mode

pip install depthfusion[mac-mlx]

Retrieval: BM25 + ChromaDB + HNSW (all tiers)
Embedding: sentence-transformers on CPU + mlx_lm.server on Apple Silicon GPU
Extras: mlx-lm, sentence-transformers, chromadb, hnswlib, fastapi, uvicorn

Best for: MacBook Pro / Mac Studio M-series solo developers who want GPU-accelerated local inference without a cloud GPU. mlx_lm serves an OpenAI-compatible endpoint on :8000.

Throughput: ~3× sentence-transformers CPU for embedding. LLM inference via mlx_lm at 40–80 tok/s on M3 Max.

4. Tool Domains — All 30+ Tools Explained

DepthFusion exposes tools via the MCP protocol (JSON-RPC 2.0 over stdio). Tools are conditionally registered based on the deployment mode and feature flags. Every tool call passes through principal binding + capability check (check_tool_access) before dispatch — unauthorized calls return an authorization error, not a generic failure.

Domain 1: Recall Core

depthfusion_recall_relevant
Primary recall entrypoint. Runs the full 4-tier cascade (BM25 → vector → reranker → HNSW) and returns ranked results with the fabric_seed score, source attribution, contradiction warnings, and backend provenance. The single most-called tool in production sessions.
depthfusion_retrieve_context
Retrieve a specific memory by ID or project scope. Used when the caller already knows which memory to fetch (e.g. after a graph traversal reveals a specific node).
depthfusion_recall_feedback
Submit positive/negative feedback on a recall result. Updates the salience score component. Good results score up; irrelevant results score down. Closes the reinforcement loop — DepthFusion gets smarter with use.

Domain 2: Capture Core

depthfusion_publish_context
Store a typed MemoryObject. The primary write operation. Runs the 3-gate materialisation policy (score threshold → novelty → capacity). Triggers Event Graph Fabric fan-out if Redis is available. Accepts type tags: decision, code_pattern, incident, discovery, note.
depthfusion_auto_learn
Extract and store key learnings from a conversation chunk automatically. Claude Code's CLAUDE.md hook triggers this after significant work sessions. Reduces manual publish burden.
depthfusion_compress_session
Summarize and store a full conversation session as a compressed memory. Useful at context window limits — preserve the essence before /compact fires.
depthfusion_tag_session
Tag the current session with metadata (project slug, agent identity, sprint tag). Used to scope subsequent recall queries. Sub-project scoping via DEPTHFUSION_WING_ID / DEPTHFUSION_ROOM_ID.
depthfusion_ingest_conversation
Import a full conversation transcript (e.g. from a Claude Code .jsonl session file) as a batch of memories. Enables post-hoc learning from completed sessions.
depthfusion_inspect_discovery
View a discovery candidate before confirming it. Inspects the proposed MemoryObject without committing to storage — useful for review gates.
depthfusion_prune_discoveries
Remove low-quality or stale discovery candidates. Manual cleanup before the autonomic daemon runs its archive pass.

Domain 3: Decisions Knowledge

depthfusion_record_decision
Store an architectural or technical decision with rationale, alternatives considered, and decision date. Creates a searchable decision log. Decisions have the highest type weight in CognitiveScorer — they surface above other memory types.
depthfusion_record_incident
Capture a bug, outage, or failure with severity, timeline, and root cause. Cross-referenced with related decisions so future agents know what went wrong after a specific architectural choice.
depthfusion_report_outcome
Record the outcome of a decision (did the chosen approach work?). Links back to the original decision via graph edge. Enables outcome-aware recall — surface decisions that had good outcomes, warn about ones that didn't.
depthfusion_mark_superseded
Mark a prior decision as superseded by a newer one. Creates a SUPERSEDES graph edge. Prevents outdated architecture docs from polluting recall. The old decision remains searchable (history) but is ranked below its successor.
depthfusion_get_cognitive_state
Return the current cognitive score breakdown for a memory or set of memories. Diagnostic tool for understanding why certain memories rank higher than others.
depthfusion_run_recursive
Trigger a recursive reasoning pass over a subset of memories — used to synthesize higher-level insights from multiple related memories.

Domain 4: Graph Event Fabric

depthfusion_graph_traverse
Navigate the Event Graph Fabric. Given a starting memory node, traverse typed edges (DERIVED_FROM, CONTRADICTS, SUPERSEDES, etc.) to find related memories. Returns the subgraph as a structured list with edge types and confidence scores.
depthfusion_graph_status
Return current graph statistics: node count, edge type distribution, most-connected nodes, last event timestamp. Health check for the Event Graph Fabric.
depthfusion_event_publish
Emit a custom event to the Redis Streams backend. Used by agent-hub orchestrators to broadcast state changes (e.g. "sprint started", "PR merged") that should be shared across the agent fleet.
depthfusion_event_seed
Seed the graph with an initial set of events (bootstrap mode). Used when migrating from another memory system or initializing a new project with historical context.
depthfusion_set_scope
Set the current principal's scope (project, wing, room). Subsequent reads and writes are filtered to this scope. Enables multi-tenant memory isolation on a single DepthFusion instance.
depthfusion_confirm_discovery
Promote a discovery candidate to a confirmed memory. Triggers materialisation pipeline. Used by the review gate pattern: an agent discovers something, a senior agent confirms it.
depthfusion_pin_discovery
Pin a memory so the eviction policy never removes it. For load-bearing facts that must persist indefinitely (e.g. "we use PostgreSQL, not MySQL").
depthfusion_agent_trail
Return the sequence of memory operations performed by a specific agent principal. Used for audit trails and debugging unexpected memory interactions.
depthfusion_set_memory_score
Manually override the cognitive score for a memory. Administrative tool — useful when the automatic scoring diverges from human judgment.

Domain 5: Project Workspace

depthfusion_register_project
Register a new project with its slug, root path, and metadata. Creates the project's memory partition. Required before scoped recall queries will return project-specific results.
depthfusion_list_projects
List all registered projects with their memory counts, last activity timestamps, and health status.
depthfusion_ingest_project
Batch-ingest a project's source files, documentation, and CLAUDE.md as memories. One-time initialization that primes DepthFusion with structural knowledge about the codebase.
depthfusion_sync_project
Incremental sync — re-ingest changed files since the last sync. Keeps DepthFusion current as the codebase evolves without a full reingest.
depthfusion_session_seed
At session start, load the top-N most relevant memories for the current project/scope. Claude Code's session-start hook calls this to pre-warm context before any user message is processed.

Domain 6: System Infrastructure

depthfusion_status
Return server health: enabled tools count, active mode, backend status, memory store stats, autonomic daemon state, Event Graph Fabric connectivity.
depthfusion_list_providers
List available retrieval backends with their status (active/inactive), tier assignments, and configuration. Shows which of BM25/Haiku/ChromaDB/HNSW are enabled in the current deployment.
depthfusion_research_topic
Synthetic tool that combines recall + graph traversal + cognitive scoring into a research pass on a topic. Returns a synthesized summary of what DepthFusion knows about a subject area.

Domain 7: Telemetry Observability

depthfusion_record_telemetry
Record model performance telemetry: tokens_in, tokens_out, latency_ms, cost_usd, quality_verdict, model_id. Feeds the budget-aware model selector's learning loop.
depthfusion_query_telemetry
Query the telemetry store — filter by model, project, date range. Returns aggregated quality_rate, avg_cost_usd, pass/fail breakdown. Powers the /api/budget-summary REST endpoint.
depthfusion_tier_status
Return per-tier retrieval latency percentiles (p50/p95), call counts, and error rates. Diagnostic for retrieval performance tuning.
depthfusion_hnsw_capability
Check if the HNSW index is built, its current size (number of vectors), and whether sentence-transformers are loaded. Gateway check before routing queries to Tier 3.
depthfusion_describe_capabilities
Self-describing tool that returns a structured list of all available tools, their domains, and tier requirements. Used by meta-agents to discover what DepthFusion can do in a given deployment.
depthfusion_surface_skill_candidates
Suggest which skills/capabilities are most relevant based on current memory corpus and recent activity patterns. Proactive recommendation engine built on top of the telemetry store.

Domain 8: Bridge v2.0.0

depthfusion_bridge
Multi-provider bridge: given a recall query, route it to multiple memory backends simultaneously (DepthFusion store + external sources) and merge results. Used by the budget-aware model selector to aggregate context from heterogeneous sources. Enables DepthFusion to act as a federation layer over multiple memory stores.

5. v2.0.0 New Features — What's Changed Since the Old Demo

The existing demo was built against v1.x. v2.0.0 ships 7 major new capabilities that must be prominently featured in the new demo. These are not incremental — they represent an architectural leap from single-user memory to enterprise-grade multi-agent cognitive infrastructure.

OIDC Authentication + RBAC

Enterprise identity integration. DepthFusion validates OIDC JWTs (Azure AD, Okta, Auth0) at the HTTP sidecar. Every tool call is bound to a Principal object — the authenticated caller identity. RBAC roles (admin, operator, reader) control which tools each principal can call.

check_tool_access(tool_name, principal) runs before every dispatch. Unauthorized calls return an authorization error, not a generic failure — the error message identifies which role is required.

Device Enrollment

New principals must complete a device enrollment flow before they can write memories. This prevents unauthorized agents from polluting shared memory stores. Enrolled devices receive a signed certificate that identifies them across sessions — enabling the agent trail audit feature.

Budget-Aware Model Selection

The recommend_model HTTP endpoint (available at the HTTP sidecar on :8765) returns ranked model candidates based on quality-per-dollar. The ranking uses 30-day rolling telemetry from depthfusion_record_telemetry — real observed quality rates, not vendor-published benchmarks.

The record_model_telemetry tool closes the feedback loop: observed outcomes (pass/fail, cost, latency) update the learning model. Each new run gets smarter recommendations.

Tauri 2 Desktop App

The app/ directory ships a React 19 + Tauri 2 + Tailwind 4 desktop client. Provides a visual memory browser, graph explorer, telemetry dashboard, and recall UI without requiring Claude Code. Built with Vite 8, TypeScript ~6.0, vitest for testing. Available for Mac, Windows, Linux.

Event Graph Fabric

Redis Streams-backed SSE pub/sub for multi-agent teams (v1.2.0, now GA in v2.0.0). Every publish/receive creates a graph node. 9 typed edge types. Enables causal lineage: which agent published what, what was derived from it, what contradicted it.

Autonomic Consolidation

Background daemon that runs near-duplicate detection and archive candidate identification every N minutes. Keeps the memory corpus clean without manual intervention. Enabled with DEPTHFUSION_AUTONOMIC=1.

ACL / Principal Authorization

Fine-grained access control at the tool level. Each tool has a required capability set. Principals are granted capabilities via the enrollment flow. The depthfusion_agent_trail tool provides full audit trails of who did what, when.

Classification-Aware Memory

7 typed MemoryObjects: Decision, CodePattern, Incident, Discovery, ProjectContext, Note, Outcome. Each type has a different CognitiveScorer weight and a different retention policy. Decisions are kept longer; Notes are evicted sooner. Type metadata is searchable.

10. Act-by-Act Storyboard Summary

Act 1 — The Problem (~30s)
Scene: Split screen. Two Claude Code terminals. Same codebase.
Agent A (left terminal): Makes architectural decision. Writes 40 lines of reasoning in its context window. Commits code. Session ends. Context wiped.
Agent B (right terminal, next day): claude code. Blank context. Starts re-deriving the same decisions. The architecture slowly diverges. A bug appears. Agent B doesn't know it was already fixed by Agent A.
Narrator text: "AI agents are amnesiac by design. Every session starts from zero. In a team of agents, this isn't just inefficiency — it's architectural drift."
Act 2 — DepthFusion in Action (~90s)
Scene: Same split screen, but now with a DepthFusion node between them.
Agent A: depthfusion_publish_context(...) — decision stored. Pipeline animation. Redis fan-out glow.
Agent A session ends. Scene: clock advances 18 hours.
Agent B starts: depthfusion_session_seed(...) — 12 memories load. The architectural decision appears in context. Agent B builds correctly on the first try.
Agent C joins: tries to store contradicting decision. ContradictionEngine fires (red glow). Both memories retained with CONTRADICTS edge. Agent B sees warning on next recall.
The Event Graph expands: nodes accumulate. Edges form a knowledge lineage. The graph is queryable. Causal history is preserved.
Narrator: "DepthFusion doesn't just remember — it understands the relationships between memories."
Act 3 — Architecture Deep Dive (~60s)
Scene: full-width architecture diagram.
Walk through each tier of the pipeline. Show the latency breakdown. Show the formula dials.
Cut to deployment mode selector: local → vps-cpu → vps-gpu → mac-mlx.
Cut to competitor matrix (3 tables, sequential reveal).
End on install command: pip install depthfusion with the GitHub star button.
